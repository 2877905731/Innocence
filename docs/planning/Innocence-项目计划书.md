## 2. 项目定位

`Innocence` 是一个面向学习、自律、陪伴和团队互助的双端软件：

- 手机端：主应用
- 电脑端：桌面挂件 / 桌面端应用
- 核心方向：数据同步、状态互通、学习陪伴、好友互动、团队协作、自律统计
- 社交定位：偏熟人圈子、轻社交，不走开放式陌生人社交路线

### 2.1 桌面端视觉方向

- 桌面版 UI 采用毛玻璃、半透明、沉浸光效风格
- 视觉参考方向：苹果系玻璃感界面、华为沉浸光效
- 该视觉要求属于全局设计约束，后续桌面挂件、桌面端首页和弹层均需遵循

## 8. 已完成模块记录

### 8.1 用户与账户基础

- 模块状态：已确认

#### 模块目标

建立一个适合双端同步、熟人圈使用、轻社交且强隐私保护的账户体系，为好友、团队、私信、学习统计和后台管理提供统一身份基础。

#### 子功能清单

- 邮箱注册
- 邮箱密码登录
- 邮箱验证码登录
- 退出登录
- 账号找回
- 头像设置
- 名称设置
- 用户号
- 多设备同时在线
- 基础资料展示
- 隐私设置
- 黑名单机制

#### 用户操作流程

1. 用户通过邮箱创建账号并登录
2. 同一个账号允许在手机端和电脑端同时在线
3. 用户完善头像和名称
4. 用户可以通过搜索用户或邀请码添加好友
5. 团队成员之间可以互相加好友，也可以只保留队友关系而不加好友
6. 用户可通过隐私设置控制资料、学习数据和私信权限
7. 用户可将不希望接触的人加入黑名单

#### 关键业务规则

- 账号体系以邮箱为主
- 邮箱登录支持两种方式并存：邮箱 + 密码、邮箱 + 验证码
- 同一个账号允许手机端和电脑端同时在线
- 邮箱仅用于注册、登录和账户找回，不作为对外搜索字段
- 搜索用户统一通过用户号、昵称、邀请码进行
- 添加好友支持两种入口：搜索用户、邀请码
- 团队成员之间支持互相加好友，但不强制要求必须加好友
- 个人资料展示信息先控制在最小范围：头像、名称、学习时长、坚持天数
- 产品阶段定位为熟人圈子，不做开放式陌生人社交
- 陌生人不可私信
- 仅好友可查看资料
- 仅队友可见学习数据
- 支持拉黑

#### 异常情况处理

- 被拉黑后，双方不可继续私信，也不可继续主动发起好友互动
- 被拉黑后，不可发送好友申请
- 被拉黑后，不可邀请对方加入团队
- 被拉黑后，不可查看对方资料
- 非好友用户尝试查看资料时，返回受限内容或空白资料页
- 非队友用户尝试查看学习数据时，返回无权限提示
- 如果团队内已有协作关系但双方未加好友，仍可保留团队协作，不强制中断

#### 与其他模块的关系

- 好友模块依赖本模块的搜索、隐私、黑名单规则
- 团队模块依赖本模块的账号身份与资料展示规则
- 私信系统依赖本模块的好友关系和陌生人私信限制
- 统计中心依赖本模块定义哪些数据允许被本人、好友、队友查看
- 系统设置会承接本模块的隐私设置、黑名单设置、登录设备管理

#### 首页 / 统计页引用方式

- 首页可引用本人学习时长、坚持天数、今日计划等本人数据
- 统计中心可展示本人完整数据
- 好友页只展示被允许公开的资料内容
- 团队页可展示队友学习数据，但权限边界必须受“仅队友可见学习数据”约束

#### 后台是否需要管理能力

- 需要
- 管理员应具备查看用户基础信息、处理举报、限制账号、封禁账号的能力
- 管理员不应随意查看私密内容，应保留必要的权限边界

#### 本模块的数据对象草案

- 用户主表 `user`
- 登录认证表 `user_auth`
- 用户资料表 `user_profile`
- 用户隐私设置表 `user_privacy_setting`
- 用户黑名单表 `user_blacklist`
- 设备会话表 `user_device_session`

#### 当前已确认结论

- 注册方式主标识为邮箱
- 邮箱登录支持两种方式并存：邮箱 + 密码、邮箱 + 验证码
- 一个账号允许手机和电脑同时在线
- 邮箱仅用于注册，不作为搜索字段暴露
- 搜索用户支持用户号、昵称、邀请码
- 添加好友支持邀请码和搜索用户
- 团队内成员可以互相加好友，也可以不加好友
- 个人资料公开信息先仅保留：头像、名称、学习时长、坚持天数
- 社交方向以熟人圈为主，不把产品做成泛社交软件
- 陌生人不可私信
- 仅好友可看资料
- 仅队友可见学习数据
- 支持拉黑
- 拉黑后的限制范围采用严格方案：不能私信、不能看资料、不能发好友申请、不能邀请进团队

### 8.2 设备与同步基础

- 模块状态：已确认

#### 模块目标

建立手机端与电脑端之间稳定、可恢复、轻量可控的数据与状态同步机制，保证桌面挂件和手机主应用在学习陪伴场景下能保持一致体验。

#### 子功能清单

- 手机与电脑双端同步
- 单账号双端同时在线
- 学习计划同步
- 学习状态同步
- 提醒状态同步
- 互动未读提示同步
- 本地离线记录
- 联网后自动补同步
- 同步冲突处理

#### 用户操作流程

1. 用户在手机端登录账号
2. 用户在电脑端登录同一账号
3. 系统允许 1 台手机和 1 台电脑同时在线
4. 用户在任一端修改学习计划、学习状态等信息
5. 另一端在联网状态下自动接收更新
6. 如果电脑端短暂断网，桌面挂件仍可先本地记录
7. 电脑端恢复联网后，自动将离线数据补同步到服务端并分发给手机端

#### 关键业务规则

- 一个账号当前阶段只允许 1 台手机 + 1 台电脑同时在线
- 手机端与电脑端需要同步的内容包括：学习状态、提醒状态、互动未读提示、学习计划
- 当前阶段不做复杂冲突合并，冲突处理规则采用“最后修改覆盖”
- 当前阶段不单独提供“设备管理”页面
- 桌面挂件支持离线短暂使用，断网期间允许先本地记录，恢复联网后自动同步

#### 异常情况处理

- 电脑端断网时，允许先本地缓存需要同步的数据
- 联网恢复后，系统自动补发未同步记录
- 手机端与电脑端同时修改同一条数据时，以最后一次提交成功的数据为准
- 如果某条离线数据同步失败，应保留重试机制，避免直接丢失
- 若超出当前允许的设备数量，应拒绝新设备登录或要求旧设备下线

#### 与其他模块的关系

- 学习计划模块依赖本模块保证双端内容一致
- 定时系统依赖本模块同步学习状态与提醒触达
- 签到系统依赖本模块同步打卡结果与坚持时长
- 互动与提醒模块依赖本模块同步未读提示与触发提醒
- 统计中心依赖本模块汇总双端产生的数据
- 系统设置会间接承接同步策略、登录状态和离线缓存相关设置

#### 首页 / 统计页引用方式

- 首页展示的今日计划、提醒、互动未读提示、学习状态需要依赖本模块实时或准实时同步
- 统计中心展示的学习时长、完成次数、坚持数据需要汇总双端数据结果
- 桌面挂件上的摘要信息应与手机端首页核心状态保持一致

#### 后台是否需要管理能力

- 需要基础管理能力
- 后台可查看用户设备在线状态、最近同步时间、同步异常次数
- 当前阶段无需给普通用户开放设备管理页面，但后台可保留排障能力

#### 本模块的数据对象草案

- 设备会话表 `user_device_session`
- 设备信息表 `user_device`
- 同步任务表 `sync_task`
- 同步日志表 `sync_log`
- 离线补传记录表 `offline_sync_record`
- 通知投递记录表 `notification_delivery`

#### 当前已确认结论

- 一个账号允许 1 台手机 + 1 台电脑同时在线
- 双端需要同步：学习状态、提醒状态、互动未读提示、学习计划
- 同一条数据发生冲突时采用“最后修改覆盖”
- 当前阶段不提供用户侧设备管理页面
- 桌面挂件支持离线记录，联网后自动同步

### 8.3 权限与内容安全基础

- 模块状态：已确认

#### 模块目标

建立一套适合熟人圈轻社交产品的基础安全治理规则，在不过度打扰正常交流的前提下，提供举报、审核、处罚、通知和简单内容拦截能力。

#### 子功能清单

- 内容举报
- 举报受理
- 管理员审核
- 删除违规内容
- 用户警告
- 用户禁言
- 用户封号
- 敏感词拦截
- 违规词替换或禁止发送
- 处罚结果通知

#### 用户操作流程

1. 用户在看到不合适内容后，可对指定内容发起举报
2. 举报进入后台待处理列表
3. 管理员查看举报内容并进行审核
4. 管理员根据违规程度采取删除内容、警告、禁言或封号
5. 被处理用户收到系统通知
6. 举报内容在处理完成后进入已处理记录，便于后续追踪

#### 关键业务规则

- 当前阶段支持举报的对象包括：团队交流内容
- 当前阶段暂不支持“用户本身”举报
- 当前阶段暂不支持“私信内容”举报
- 内容审核方式采用“先发出，用户举报后管理员处理”
- 管理员可采用的处罚手段包括：删除内容、警告、禁言、封号
- 发送内容时启用简单敏感词拦截
- 对违规词采用替换或禁止发送两种策略
- 用户被举报处理或被处罚后，需要收到系统通知

#### 异常情况处理

- 已被删除的违规内容应保留后台审计记录，避免处理后无法追溯
- 对被多次举报但尚未处理的内容，可在后台标记高优先级
- 敏感词命中时，前端应明确提示是被替换还是禁止发送
- 被禁言用户仍可登录和查看内容，但不能继续发送受限内容
- 被封号用户不可继续登录系统

#### 与其他模块的关系

- 团队模块中的团队交流会使用本模块的举报与审核规则
- 系统通知模块会承接处罚结果通知
- 后台管理模块会承接举报列表、审核操作、处罚记录
- 用户与账户基础模块会承接禁言、封号等账户级限制

#### 首页 / 统计页引用方式

- 团队相关页面中的交流区需要遵循同样的审核和举报规则
- 统计页原则上不直接展示违规处理内容，只展示正常业务数据

#### 后台是否需要管理能力

- 需要，且是核心能力之一
- 后台需具备举报列表、内容查看、违规判定、处罚执行、处理记录查询能力
- 后台需保留敏感词管理入口或至少预留配置能力

#### 本模块的数据对象草案

- 举报主表 `report_record`
- 举报内容关联表 `report_target`
- 审核处理记录表 `report_audit_record`
- 处罚记录表 `punishment_record`
- 敏感词表 `sensitive_word`
- 内容审核日志表 `content_security_log`

#### 当前已确认结论

- 举报对象当前只支持：团队交流内容
- 当前阶段不支持“用户本身”举报
- 当前阶段不支持“私信内容”举报
- 内容治理方式采用“先发后审，用户举报后管理员处理”
- 管理员处罚方式包括：删除内容、警告、禁言、封号
- 内容发送时启用简单敏感词拦截
- 对违规词采用“替换或禁止发送”策略
- 被举报处理或被处罚的用户需要收到系统通知

#### 当前待确认问题

- 暂无，本模块已封板
- 后续可扩展项：如产品后期出现私信骚扰场景，可再补充“私信举报”能力

### 8.4 好友模块

- 模块状态：已确认

#### 模块目标

建立一个符合熟人圈定位的好友关系系统，支持用户通过申请建立连接、按分组管理关系，并在断开关系时彻底清除互动痕迹。

#### 子功能清单

- 搜索添加好友
- 邀请码添加好友
- 好友申请
- 好友申请同意 / 拒绝
- 删除好友
- 好友分组
- 自定义分组
- 好友上限控制

#### 用户操作流程

1. 用户通过用户号、昵称或邀请码发起加好友申请
2. 对方收到好友申请后，可选择同意或拒绝
3. 申请通过后，双方建立好友关系
4. 用户可对好友进行自定义分组管理

2. 用户通过拖动长条跨越多个半小时单位，形成一个学习时段

### 8.8 定时系统

- 模块状态：已确认

#### 模块目标

建立一个围绕“即时开始学习”的轻量执行系统，让用户可以按结束时间开启学习时长计时，并自由组合番茄循环，用来支撑真实学习过程中的专注、休息和结果统计。

#### 子功能清单

- 学习时长模式
- 结束时间选择
- 实时显示学习时长
- 番茄模式
- 番茄循环自定义
- 学习时长与番茄联动
- 提前结束
- 后台持续计时
- 锁屏持续计时
- 断网持续计时

#### 用户操作流程

1. 用户想开始学习时，先设置一个结束时间
2. 系统在用户选择结束时间时，实时显示对应的学习时长
3. 用户可直接开启学习时长模式，作为一次即时学习计划
4. 用户也可以额外开启番茄模式，并自定义每轮学习时长和休息时长
5. 如果用户选择将番茄与学习时长关联，番茄循环会一直运行到学习时长结束
6. 如果用户手动提前结束学习，本次学习时长和番茄循环都会一起结束
7. 即使应用切到后台、锁屏或短暂断网，计时仍继续

#### 关键业务规则

- 学习时长模式采用“设置结束时间”的方式启动
- 用户在选择结束时间时，界面需边选边显示本次学习时长
- 学习时长本质上属于一次即时计划，用户想学时即可发起
- 番茄模式支持完全自定义学习时长和休息时长
- 番茄模式可选择是否与学习时长关联
- 当番茄与学习时长关联时，番茄循环次数不可单独设置
- 当番茄与学习时长关联时，番茄默认一直循环到学习时长结束
- 当番茄未与学习时长关联时，可在后续实现中继续保留循环次数配置空间
- 用户手动提前结束学习时，学习时长计时和番茄循环都同时结束
- 定时过程中切后台、锁屏、断网时，计时继续

#### 异常情况处理

- 若用户设置的结束时间早于当前时间，应禁止启动并提示重新选择
- 若番茄与学习时长关联后仍尝试设置循环次数，应直接禁用该配置入口
- 若设备短暂断网，计时结果先本地持续记录，恢复联网后再同步
- 若应用被系统异常回收，重新打开后应尽量恢复本次计时状态或恢复结果记录
- 若用户在学习结束前手动终止，不应继续累计后续学习时长

#### 与其他模块的关系

### 8.9 签到系统

#### 模块目标

5. 若当天仅有学习时长但未完成签到标准，则保留学习时长，但不计为签到成功
6. 未达到签到标准的情况进入失败记录

- 若当天学习了但未达到签到标准，学习时长仍保留
- 未达签到标准的当天记录进入失败记录
- 当前阶段不支持补签

#### 异常情况处理

- 用户未完成当天计划时点击签到，应明确提示未满足签到条件
- 当天有学习时长但签到失败时，不应丢失学习时长数据
- 连续坚持中断时，应保留累计坚持天数，但连续坚持天数重新计算
- 若用户误操作提前签到失败，当天后续完成计划后仍可再次尝试签到

- 签到失败记录表 `check_in_fail_record`

#### 当前已确认结论

### 8.14 后台管理

#### 模块目标

建立一个简洁但够用的管理后台，用于处理用户、团队、通知、举报和基础运营概览，保证前台社区秩序、团队关系和通知投递都能被有效管理。

3. 管理员进入团队管理查看团队并在必要时移除成员或强制解散团队
4. 管理员进入通知管理发布系统公告，或定向通知某个用户、某个团队

- 定向通知发送失败时，应记录失败原因并支持重试
- 后台首页若某些统计暂时无数据，应正常显示空状态而非报错

- 团队模块提供团队关系、成员和解散对象

- 权限与内容安全基础模块提供举报、处罚和复查来源

- 后台首页只做管理概览，不复用前台首页交互逻辑

### 8.15 首页

- 模块状态：已确认

#### 模块目标

将学习执行、签到反馈、统计趋势、备忘录摘要和队友动态聚合到一个高频入口页面中，让用户一打开应用或桌面挂件，就能快速进入状态并看到最重要的信息。

#### 子功能清单

- 顶部重点区
- 当前学习状态
- 当前番茄 / 学习倒计时
- 今日计划
- 签到状态
- 趋势图摘要
- 队友动态摘要
- 备忘录摘要
- 关键统计摘要

#### 用户操作流程

1. 用户进入首页后，优先看到当前学习状态和今日计划
2. 若当前存在正在进行中的学习或番茄任务，首页顶部重点区展示倒计时和预计结束时间
3. 用户向下查看签到状态、趋势图摘要、关键统计摘要和备忘录摘要
4. 用户可在首页查看一个轻量的队友动态摘要区
5. 用户可点击各摘要区进入对应详细页面

#### 关键业务规则

- 首页采用“上方重点区 + 下方多卡片区”的布局结构
- 首页顶部重点展示：当前学习状态 + 今日计划
- 首页需要显示当前正在进行中的番茄 / 学习倒计时
- 队友动态在首页中只占一个小摘要区，不做过重展示
- 备忘录在首页中只展示前几条摘要，不直接展示完整内容
- 首页趋势图可复用统计中心数据源
- 首页需聚合签到状态、坚持数据、学习状态、备忘录摘要和队友动态摘要
- 桌面挂件首页同样遵循该信息优先级，但展示更克制、更轻量
- 桌面版首页与挂件需服从毛玻璃、半透明、沉浸光效全局视觉约束

#### 异常情况处理

- 当用户当前没有进行中的学习任务时，顶部重点区应回退为“今日计划”主导视图
- 当某一模块暂无数据时，应显示空状态占位，而不是让首页布局塌陷
- 若桌面挂件空间有限，应优先保留当前学习状态、今日计划、倒计时和摘要信息

#### 与其他模块的关系

- 学习计划模块提供今日计划内容
- 定时系统提供当前学习状态、番茄状态和倒计时
- 签到系统提供签到状态、连续坚持天数、累计坚持天数和累计坚持时长
- 统计中心提供趋势图和关键统计摘要
- 备忘录模块提供前几条备忘录摘要
- 团队模块和互动模块提供队友动态摘要
- 系统设置中的挂件选项决定首页 / 挂件展示模块

#### 首页 / 统计页引用方式

- 本模块本身即首页聚合页
- 首页只做摘要展示，详细数据进入统计中心、计划页、备忘录页等子页面查看

#### 后台是否需要管理能力

- 首页本身不需要后台管理能力
- 仅依赖后台提供的系统公告、账号状态和团队治理结果间接影响展示内容

#### 本模块的数据对象草案

- 首页不单独增加核心业务主表
- 首页主要依赖其他模块的数据聚合结果

#### 当前已确认结论

- 首页最上方优先突出：当前学习状态 + 今日计划
- 首页采用：上方重点区 + 下方多卡片区
- 队友动态在首页中只做小摘要区
- 备忘录在首页中展示前几条摘要，点击进入详情
- 首页显示当前正在进行中的番茄 / 学习倒计时
- 桌面端首页和挂件遵循毛玻璃、半透明、沉浸光效风格
