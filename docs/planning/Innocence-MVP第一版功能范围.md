# Innocence MVP 第一版功能范围

## 1. 文档说明

- 文档目标：明确 `Innocence` 第一版真正要做的范围，避免开发初期功能过多、节奏失控。
- 使用方式：
  - 作为开发排期的基础
  - 作为前后端联调范围的边界
  - 作为后续“第一版完成标准”的判断依据
- 当前结论类型：第一版范围确认稿

## 2. MVP 的核心目标

`Innocence` 第一版不是要一次把所有设想都做满，而是先把最核心的产品价值稳定跑通：

1. 用户可以在手机端和电脑端使用同一个账号，并实现基础数据同步。
2. 用户可以完成“计划 - 学习 - 记录 - 签到 - 统计”的个人自律闭环。
3. 用户可以和少量熟人好友、队友建立轻量陪伴关系，而不是做开放式社交。
4. 桌面端要具备明确差异化体验，至少完成桌面挂件 / 桌面首页的毛玻璃半透明基础视觉。
5. 后台要能支持最基本的内容治理和账号治理，保证产品可控。

## 3. 第一版要验证的产品价值

### 3.1 个人价值

- 今天学什么，可以提前安排。
- 正在学习时，可以专注计时。
- 学完之后，有记录、有统计、有签到反馈。

### 3.2 陪伴价值

- 可以和好友、队友保持轻量连接。
- 可以看到队友今天的计划进度和学习时长。
- 可以提醒队友，也能收到队友完成学习后的通知。

### 3.3 双端价值

- 手机端适合完整操作。
- 桌面端适合陪伴式展示和快速进入学习状态。
- 两端核心数据保持一致，不出现明显割裂。

## 4. 第一版目标用户

第一版更适合这类用户：

- 有学习、自律、备考、长期坚持需求的人
- 更偏向熟人陪伴，而不是陌生人社交的人
- 需要桌面挂件陪伴感的人
- 希望用较轻方式和 1 到 5 人小团队共同坚持的人

第一版不以这些方向为重点：

- 泛社交
- 陌生人匹配
- 复杂社区互动
- 大型团队协作

## 5. 第一版必须跑通的核心闭环

### 5.1 个人闭环

1. 注册登录
2. 创建今日学习计划
3. 开始学习时段 / 番茄循环
4. 完成任务进度
5. 手动签到
6. 查看统计结果

### 5.2 团队闭环

1. 创建团队或加入团队
2. 查看队友今日学习进度
3. 提醒队友
4. 队友完成后收到通知
5. 团队群聊中进行简单交流

### 5.3 双端闭环

1. 手机端登录
2. 桌面端登录
3. 计划、学习状态、通知红点同步
4. 桌面端展示今日计划和当前学习状态

## 6. 第一版功能范围总原则

### 6.1 要做的方向

- 做完整闭环
- 做稳定同步
- 做轻社交陪伴
- 做最小但够用的后台治理

### 6.2 刻意收缩的方向

- 不追求复杂社交玩法
- 不追求过度花哨的功能堆叠
- 不追求第一版就把后台做得很重
- 不追求复杂多设备体系

### 6.3 第一版判断标准

不是“页面多不多”，而是下面这些事情能不能稳定完成：

- 用户能不能顺利开始今天的学习
- 用户能不能顺利完成今天的签到
- 队友能不能顺利看到彼此进度并互相提醒
- 手机和电脑能不能保持核心数据一致

## 7. 第一版明确纳入范围的模块

### 7.1 账户与登录

第一版必须做：

- 邮箱注册
- 邮箱 + 密码登录
- 邮箱 + 验证码登录
- 找回密码
- 头像、昵称设置
- 用户号生成
- 账号注销
- 黑名单
- 隐私设置

第一版简化方式：

- 邮箱只用于注册、登录、找回，不对外展示
- 公开资料只保留最基础内容：
  - 头像
  - 昵称
  - 学习时长
  - 坚持天数

### 7.2 设备与同步

第一版必须做：

- 同一账号支持 `1 台手机 + 1 台电脑` 同时在线
- 学习状态同步
- 学习计划同步
- 通知红点同步
- 桌面端临时离线后恢复同步

第一版简化方式：

- 冲突处理采用“最后修改覆盖”
- 不单独做复杂设备管理页
- 不做多电脑、多手机并发

### 7.3 好友模块

第一版必须做：

- 搜索用户
- 通过用户号 / 昵称 / 邀请码发起好友申请
- 同意 / 拒绝好友申请
- 好友列表
- 删除好友
- 好友分组
- 拉黑

第一版必须遵守：

- 好友申请必须经过同意
- 好友上限 `200`
- 删除好友后，私信记录一起删除
- 陌生人不可私信
- 仅好友可看资料

### 7.4 团队模块

第一版必须做：

- 创建团队
- 生成团队邀请码
- 通过邀请码申请加入团队
- 直接邀请指定用户加入团队
- 同意 / 拒绝团队邀请
- 查看团队成员列表
- 队长移除成员
- 队长解散团队

第一版必须遵守：

- 一个人只能加入 `1` 个团队
- 团队人数上限 `5`
- 只有队长可以移除成员
- 所有团队邀请都需要同意
- 团队解散后，团队相关交流数据直接删除

### 7.5 私信、团队交流、提醒

第一版必须做：

- 好友私信
- 团队群聊
- 提醒某个队友
- 提醒整个团队
- 未读红点
- 举报聊天内容

第一版简化方式：

- 私信只做文字
- 团队群聊只做文字
- 不做图片、语音、文件
- 不做复杂消息状态体系

第一版必须遵守：

- 私信只允许好友之间发送
- 团队交流内容先发出，再由举报进入后台处理
- 敏感词先做基础拦截和替换 / 禁发

### 7.6 学习计划

第一版必须做：

- 短计划
- 长计划
- 超长计划
- 计划进度清单
- 保存短计划模板
- 套用短计划模板
- 失败记录

第一版简化方式：

- 短计划按天，以半小时单位拖动排布
- 长计划按周，重点支持套用短计划模板
- 超长计划按天规划任务
- 失败记录先做列表，不做复杂分类

### 7.7 定时系统

第一版必须做：

- 学习时段模式
- 番茄模式
- 学习时段与番茄绑定
- 当前学习状态展示
- 历史学习记录

第一版必须遵守：

- 用户通过结束时间开始学习
- 前端需要实时显示学习总时长
- 番茄模式完全自定义
- 绑定学习时段后，番茄一直循环到学习时段结束
- 用户手动结束时，两者同时结束

### 7.8 签到系统

第一版必须做：

- 手动签到
- 连续签到天数
- 累计签到天数
- 累计学习时长展示
- 签到失败记录

第一版明确不做：

- 自定义签到任务
- 补签

第一版必须遵守：

- 当天必须有计划
- 用户必须手动点击签到
- 当天计划完成后才能签到成功
- 如果只学习但未满足签到条件，保留学习时长，但签到记失败

### 7.9 统计中心

第一版必须做：

- 番茄完成次数
- 学习总时长
- 签到天数
- 计划完成率
- 签到成功率
- 趋势图
- 失败次数摘要
- 队友学习摘要

第一版简化方式：

- 趋势图用一张图切换近 `7 天` 和近 `30 天`
- 不做复杂排名
- 队友信息以轻量展示为主

### 7.10 通知中心

第一版必须做：

- 好友申请通知
- 团队邀请通知
- 队友提醒通知
- 队友完成通知
- 签到结果通知
- 系统公告通知
- 已读 / 未读
- 通知内直接处理好友申请和团队邀请

第一版简化方式：

- 通知保留最近 `30 天`
- 手机端做推送
- 桌面端做系统通知
- 不做过复杂的通知订阅组合

### 7.11 备忘录

第一版必须做：

- 文本备忘录
- 清单备忘录
- 卡片式展示
- 同步到桌面挂件

第一版明确不做：

- 备忘录提醒
- 回收站

### 7.12 系统设置

第一版必须做：

- 账户资料设置
- 隐私设置
- 通知设置
- 同步相关设置
- 外观设置
- 桌面挂件设置
- 清理缓存
- 退出登录
- 注销账号

第一版必须遵守：

- 支持浅色 / 深色模式
- 通知开关按类型拆分
- 桌面挂件设置至少包含：
  - 开机自启
  - 置顶显示
  - 显示哪些模块

### 7.13 首页与桌面挂件

第一版必须做：

- 首页顶部当前学习状态区
- 今日计划摘要
- 签到摘要
- 趋势图摘要
- 队友动态摘要
- 备忘录摘要
- 桌面挂件摘要视图

第一版简化方式：

- 首页做聚合视图，不堆太多重交互
- 备忘录只显示少量摘要
- 队友动态做小型摘要区

第一版视觉底线：

- 桌面端必须体现毛玻璃、半透明、柔和沉浸光效
- 第一版先做基础视觉实现，不要求一开始就堆很多复杂动效

### 7.14 后台管理

第一版必须做：

- 管理员登录
- 用户查询
- 用户禁言
- 用户封号
- 团队查询
- 强制解散团队
- 移除团队成员
- 系统公告发布
- 定向通知用户
- 定向通知团队
- 举报列表
- 举报处理记录
- 举报复核
- 敏感词管理

第一版简化方式：

- 后台首页先做简单摘要看板
- 不做复杂 BI
- 不做复杂审批流

## 8. 第一版明确暂不纳入范围

这些功能不是不要，而是第一版先不做，避免把项目拉得太长：

- 陌生人私信
- 陌生人社交广场
- 推荐好友 / 推荐团队
- 图片、语音、文件聊天
- 团队留言板
- 好友留言板
- 自定义签到任务
- 补签
- 多手机、多电脑同时在线
- 复杂同步冲突合并
- 备忘录提醒
- 备忘录回收站
- 复杂排行榜系统
- 复杂勋章、成就体系
- AI 内容审核
- 多层级管理员权限体系

## 9. 第一版建议拆分的开发优先级

### 9.1 第一阶段：先做底层可运行

- 账户与登录
- 设备与同步
- 系统设置基础
- 后台管理员登录与基础权限

目标：

- 用户能登录
- 双端能同步
- 后台能进

### 9.2 第二阶段：做社交骨架

- 好友
- 团队
- 私信
- 团队群聊
- 通知基础

目标：

- 熟人关系建立起来
- 小团队陪伴关系跑通

### 9.3 第三阶段：做学习主闭环

- 学习计划
- 定时系统
- 签到系统
- 备忘录
- 首页聚合

目标：

- 用户能真正完成一天的学习闭环

### 9.4 第四阶段：做结果与治理

- 统计中心
- 举报处理
- 敏感词管理
- 公告与定向通知

目标：

- 产品具备基础运营能力和安全能力

### 9.5 第五阶段：做桌面体验强化

- 桌面挂件优化
- 桌面视觉强化
- 桌面通知联动

目标：

- 桌面端形成明显差异化体验

## 10. 第一版完成标准

满足下面这些条件，可以认为 `Innocence` 第一版达到了可上线测试的状态：

1. 用户可以完成注册、登录、资料设置、隐私设置。
2. 手机端和桌面端可以同时在线，并同步核心数据。
3. 用户可以创建计划、开始学习、结束学习、查看记录。
4. 用户可以完成签到，并能看到连续天数、累计天数、累计时长。
5. 用户可以加好友、组团队、看队友进度、提醒队友。
6. 用户可以进行基础私信和团队群聊。
7. 首页和桌面挂件可以正常展示关键摘要信息。
8. 后台可以处理举报、处罚用户、发布公告。
9. 整体权限边界符合已经确认的隐私规则。

## 11. 当前结论

- `Innocence` 第一版不是做“大而全”，而是做“闭环完整、双端可用、轻社交陪伴成立”。
- 只要把本文范围稳定做完，就已经是一个具备明显产品形态的第一版，而不是功能半成品。
- 下一步最适合继续输出的文档是：
  - `开发排期草案`
  - `后端模块拆分建议`
  - `前端页面与接口映射表`
